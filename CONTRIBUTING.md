## Кодекс поведения

Будьте уважительны и конструктивны. Обсуждаем идеи и код, а не людей.
Любые формы харассмента недопустимы.

## Как сообщить о баге

Перед созданием issue:

1. Проверьте, что баг ещё не заведён в [Issues](../../issues).
2. Убедитесь, что воспроизводится на актуальном `main`.

В issue укажите:

- что произошло и что ожидалось;
- шаги воспроизведения (соперник, действия, последовательность фигур);
- браузер и ОС;
- скриншот или запись, если уместно;
- ошибки из консоли DevTools.

## Как предложить улучшение

Откройте issue с тегом `enhancement`: опишите проблему, которую решает идея,
и предлагаемое поведение. Для крупных изменений лучше обсудить подход в issue
**до** написания кода.

## Локальная разработка

Нужен Node.js 18+ и npm.

```bash
git clone https://github.com/art-ps/tetris-rl-web.git
cd tetris-rl-web
npm install
npm run dev       # http://localhost:5173
```

Перед каждым коммитом прогоняйте проверки:

```bash
npm test          # vitest: сверка порта с эталоном tetris-rl
npm run build     # tsc --noEmit + сборка vite (типы и бандл должны проходить)
```

### Важно: паритет с tetris-rl

Логика игры (`src/game/`) — это **порт** оригинального проекта
[tetris-rl](https://github.com/art-ps/tetris-rl). Тесты в
`src/game/__tests__/port.test.ts` сверяют фичи, Q-значения и валидные действия
с эталоном `fixtures.json`, сгенерированным из Python-реализации.

Если ваше изменение затрагивает правила игры, фичи состояния или forward-pass
сети — **тесты должны остаться зелёными**. Не редактируйте `fixtures.json`
вручную: его пересоздаёт `python scripts/gen_fixtures.py` из tetris-rl.

## Pull request

1. Сделайте форк и создайте ветку от `main`:
   `git checkout -b feature/краткое-описание`.
2. Делайте сфокусированные коммиты — одно логическое изменение на коммит.
3. Сообщения коммитов: повелительное наклонение, по-английски, ≤ 72 символа
   в заголовке (например, `Add soft-drop scoring`). Тело — при необходимости
   объяснить «почему».
4. Убедитесь, что `npm test` и `npm run build` проходят.
5. Откройте PR в `main`, опишите изменение и свяжите соответствующий issue
   (`Closes #123`).
6. Держите PR небольшим и по теме — так его проще ревьюить.

## Стиль кода

- TypeScript строгий; новый код типизирован, без `any` без нужды.
- Следуйте существующему стилю файла — не переформатируйте чужой код.
- Минимум зависимостей: новые npm-пакеты добавляются только при явной
  необходимости и обосновываются в PR.
- Игровая логика (`src/game/`, `src/ai/`) не должна зависеть от React.
- Комментарии — по делу, объясняют «почему», а не «что».
